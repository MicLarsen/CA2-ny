The REST-api (REpresentational State Transfer)

We decided to make three different REST classes, so we could keep a relatively loose coupling and high cohesion.
Relativelit loose coupling since REST is a decoupling arcitectual design that takes less bandwith than SOAP (Simple Object Access Protocol).
REST typically runs over HTTP have a few architectual contraints which is:

  • Decouples consumers from producers
  • Stateless existence, which means that each packets travels without reference to other packages in the same way as the IP
  • Able to leverage a cache
  • Leverages a layered system
  • Leverages a uniform interface

PersonRest:

Here we handle all the request concerning Persons.

  • getPersonById(int id)
  • getPersonByPhone(int phoneNum)
  • getAllPersonWithHobby(Hobby hobby)
  • getPersonsByZip(int zipCode)
  • getHobbyCount(Hobby hobby)
  • deletePerson(int id)
  • editPerson(int id)
  • addPerson(Person person)

CompanyRest:

Here we handle all the request concerning Company.

  • getCompanyByCVR(int cvr)
  • getCompanyByPhone(int phone)
  • getCompaniesWithXOrMoreEmpl(int num);
  • deleteCompany(int cvr);
  • editCompany(Company comp);
  • createCompany(Company comp);
  • editPerson(int id)
  • addPerson(Person person)
